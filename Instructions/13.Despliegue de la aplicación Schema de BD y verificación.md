# Despliegue de la aplicación: Schema de BD y verificación
[[1.DevOps-Proyect-01]][[Infra DevOps]][[AWS]]

El RDS está en subnet privada — **no es accesible desde tu máquina local directamente**. Debes conectarte via Bastion.

```bash
# Paso 1 — Obtener la IP del Bastion dinámicamente
IP_BASTION=$(aws ec2 describe-instances \
    --filters "Name=tag:Name,Values=BastionHost" \
              "Name=instance-state-name,Values=running" \
    --query "Reservations[0].Instances[0].PublicIpAddress" \
    --output text --region us-east-1)
echo "Bastion IP: $IP_BASTION"
```

```bash
# Si el SSH da timeout, tu IP puede haber cambiado — actualiza el BastionSG

MY_IP=$(curl -s https://checkip.amazonaws.com)
SG_BASTION_ID=$(aws ec2 describe-security-groups \
    --filters "Name=group-name,Values=BastionSG" \
    --query "SecurityGroups[0].GroupId" --output text --region us-east-1)
    
# Revocar IP anterior y agregar la actual
aws ec2 revoke-security-group-ingress \
    --group-id $SG_BASTION_ID --protocol tcp --port 22 \
    --cidr IP_ANTERIOR/32 --region us-east-1

aws ec2 authorize-security-group-ingress \
    --group-id $SG_BASTION_ID --protocol tcp --port 22 \
    --cidr $MY_IP/32 --region us-east-1
echo "Regla SSH actualizada para IP: $MY_IP"
```

```bash
# Paso 2 — Conectarse al Bastion con agent forwarding (necesario para saltar a instancias privadas)

ssh-add ~/.ssh/DevOpsKeyPair.pem
ssh -A -i ~/.ssh/DevOpsKeyPair.pem ec2-user@$IP_BASTION
```

```bash
# Paso 3 — Dentro del Bastion: instalar mysql client y conectarse al RDS

sudo yum install -y mysql

RDS_ENDPOINT=$(aws rds describe-db-instances \
    --db-instance-identifier prod-mysql \
    --query "DBInstances[0].Endpoint.Address" \
    --output text --region us-east-1)

# ⚠️ Asegúrate de dejar espacio entre el endpoint y -u
mysql -h $RDS_ENDPOINT -u admin -p
```

> ⚠️ **Si la conexión a MySQL tarda mucho o da timeout** desde el Bastion, el problema es que el `DatabaseSG` no tiene una regla que permita tráfico desde el `BastionSG`. Agrégala desde tu máquina local:

```bash
 SG_BASTION_ID=$(aws ec2 describe-security-groups \
     --filters "Name=group-name,Values=BastionSG" \
     --query "SecurityGroups[0].GroupId" --output text --region us-east-1)
 
 SG_DATABASE_ID=$(aws ec2 describe-security-groups \
     --filters "Name=group-name,Values=DatabaseSG" \
     --query "SecurityGroups[0].GroupId" --output text --region us-east-1)
 ``
 aws ec2 authorize-security-group-ingress \
     --group-id $SG_DATABASE_ID \
     --protocol tcp --port 3306 \
          --source-group $SG_BASTION_ID \
     --region us-east-1


```

## Crear la base de datos en RDS

> ⚠️ **Este paso es obligatorio.** La app espera encontrar una base de datos llamada `UserDB` en el RDS. Si no existe, Tomcat arrancará pero fallará al intentar conectarse y la app no funcionará.

Una vez conectado a MySQL desde el Bastion (paso anterior), ejecuta:

```sql
-- Ver las bases de datos existentes
SHOW DATABASES;

-- Crear la base de datos con el nombre exacto que usa application.properties
CREATE DATABASE IF NOT EXISTS UserDB;

-- Verificar que se creó
SHOW DATABASES;
-- Debe aparecer "UserDB" en la lista

EXIT;

```
> ℹ️ **¿Por qué solo la BD y no las tablas?** La app usa `spring.jpa.hibernate.ddl-auto = update` en el `application.properties`. Esto le indica a Hibernate que cree o actualice las tablas automáticamente al arrancar Tomcat. Solo necesitas crear la base de datos vacía — las tablas (`users`, etc.) las genera Hibernate solo en el primer arranque.

> ℹ️ **Nombre de la BD:** El nombre `UserDB` debe coincidir exactamente con el que aparece en la URL del `application.properties`:

```properties
spring.datasource.url = jdbc:mysql://ENDPOINT:3306/UserDB
```

> Si cambias el nombre aquí, debes cambiarlo también en el `application.properties`.

## Verificar el WAR en las instancias Tomcat

Las instancias del ASG descargan el WAR desde JFrog al arrancar via `tomcat-userdata.sh`. **Antes de considerar el despliegue completo**, verifica que el WAR se descargó correctamente:

```bash
# Conectarse a una instancia Tomcat via Bastion
ssh -A -i ~/.ssh/DevOpsKeyPair.pem ec2-user@$IP_BASTION
ssh ec2-user@TOMCAT_PRIVATE_IP

# Verificar que el WAR existe y tiene el tamaño correcto (~37MB)
ls -lh /opt/tomcat/webapps/ROOT.war
# ⚠️ Si pesa solo ~12KB es una página HTML de error — el download falló

# Verificar integridad del WAR
unzip -t /opt/tomcat/webapps/ROOT.war | tail -3
# Debe decir: No errors detected

# Verificar que se desplegó (debe existir la carpeta ROOT/)
ls /opt/tomcat/webapps/ROOT/
# Debe mostrar: META-INF  pages  WEB-INF

# Ver logs de arranque
sudo tail -30 /opt/tomcat/logs/catalina.out
```

**Si el WAR está corrupto o es una página HTML** (esto ocurre cuando JFrog requiere autenticación y el wget no tiene credenciales válidas), reemplázalo manualmente:

```bash
# Opción A — Descargar directamente en la instancia con credenciales
sudo rm -f /opt/tomcat/webapps/ROOT.war
sudo rm -rf /opt/tomcat/webapps/ROOT
sudo wget --user=TU_JFROG_EMAIL --password=TU_JFROG_TOKEN \
    -O /opt/tomcat/webapps/ROOT.war \
    "https://devopsproyect01.jfrog.io/artifactory/libs-release-local/com/devopsrealtime/dptweb/1.0/dptweb-1.0.war"
ls -lh /opt/tomcat/webapps/ROOT.war  # debe ser ~37MB

# Opción B — Subir desde tu máquina local via SCP
# Desde tu máquina local:
curl -u TU_EMAIL:TU_TOKEN \
    -o ~/dptweb-1.0.war \
    "https://devopsproyect01.jfrog.io/artifactory/libs-release-local/com/devopsrealtime/dptweb/1.0/dptweb-1.0.war"
unzip -t ~/dptweb-1.0.war | tail -3  # verificar integridad
scp -i ~/.ssh/DevOpsKeyPair.pem ~/dptweb-1.0.war ec2-user@$IP_BASTION:~/
# Desde el Bastion:
scp ~/dptweb-1.0.war ec2-user@TOMCAT_PRIVATE_IP:~/
# Dentro de la instancia Tomcat:
sudo systemctl stop tomcat
sudo rm -f /opt/tomcat/webapps/ROOT.war
sudo rm -rf /opt/tomcat/webapps/ROOT
sudo cp ~/dptweb-1.0.war /opt/tomcat/webapps/ROOT.war
sudo systemctl start tomcat
```

## Verificar y corregir application.properties

El `application.properties` dentro del WAR debe apuntar al endpoint correcto de RDS. Si la app muestra error de Hibernate o la página de Tomcat default en lugar de la app, el problema suele ser el endpoint incorrecto.

```bash
# Ver el endpoint actual configurado
cat /opt/tomcat/webapps/ROOT/WEB-INF/classes/application.properties

# Si el endpoint es incorrecto, obtener el real
RDS_ENDPOINT=$(aws rds describe-db-instances \
    --db-instance-identifier prod-mysql \
    --query "DBInstances[0].Endpoint.Address" \
    --output text --region us-east-1)
echo $RDS_ENDPOINT

# Probar conectividad desde la instancia Tomcat antes de editar
timeout 5 bash -c "echo > /dev/tcp/$RDS_ENDPOINT/3306" \
    && echo "CONECTADO" || echo "FALLO — revisar SecurityGroups"

# Editar application.properties con el endpoint correcto
sudo nano /opt/tomcat/webapps/ROOT/WEB-INF/classes/application.properties
```

El archivo debe contener exactamente:

```properties
spring.datasource.url = jdbc:mysql://ENDPOINT_RDS.us-east-1.rds.amazonaws.com:3306/UserDB
spring.datasource.username = admin
spring.datasource.password = TU_PASSWORD_RDS
spring.datasource.driver-class-name = com.mysql.cj.jdbc.Driver
spring.jpa.hibernate.ddl-auto = update
```

> ⚠️ El `/UserDB` al final de la URL es **obligatorio** — es el nombre de la base de datos a la que Tomcat se conecta. Sin él, la conexión falla.

```bash
# Reiniciar Tomcat para aplicar cambios
sudo systemctl restart tomcat

# Verificar que arrancó sin errores de conexión
sudo tail -30 /opt/tomcat/logs/catalina.out | grep -i "error\|exception\|started"

# Si no hay errores, probar la app localmente
curl http://localhost:8080/
curl http://localhost:8080/pages/login.jsp
```

> ℹ️ **Solución definitiva:** Los cambios manuales al `application.properties` no sobreviven a un redeploy. La solución correcta es actualizar el endpoint en el **código fuente** (`src/main/resources/application.properties` en la instancia Maven), hacer un nuevo build con `mvn clean deploy -DskipTests -s settings.xml`, y luego redeplegar el WAR generado.

## Verificar SecurityGroups de las instancias del ASG

Las instancias lanzadas por el ASG deben tener asignados los SecurityGroups correctos (`BackendSG` para Tomcat, `FrontendSG` para Nginx). Si se asignó el SG `default` por error, la instancia no podrá comunicarse con RDS ni con el NLB.

```bash
# Verificar qué SG tiene una instancia
curl http://169.254.169.254/latest/meta-data/security-groups
# Debe mostrar: BackendSG (para Tomcat) o FrontendSG (para Nginx)
# Si muestra "default" → hay que corregirlo

# Desde tu máquina local — corregir el SG de una instancia
TOMCAT_INSTANCE_ID=$(aws ec2 describe-instances \
    --filters "Name=private-ip-address,Values=TOMCAT_PRIVATE_IP" \
    --query "Reservations[0].Instances[0].InstanceId" \
    --output text)

BACKEND_SG_ID=$(aws ec2 describe-security-groups \
    --filters "Name=group-name,Values=BackendSG" \
    --query "SecurityGroups[0].GroupId" \
    --output text)

aws ec2 modify-instance-attribute \
    --instance-id $TOMCAT_INSTANCE_ID \
    --groups $BACKEND_SG_ID
echo "SG corregido a BackendSG"
```

> ⚠️ Si el Launch Template tiene el SG incorrecto, **todas las instancias nuevas** del ASG también tendrán el problema. Verifica el Launch Template y crea una nueva versión si es necesario:

```bash
 # Verificar SG en el Launch Template
aws ec2 describe-launch-template-versions \
   --launch-template-name TomcatLT \
    --query "LaunchTemplateVersions[0].LaunchTemplateData.SecurityGroupIds"
```
